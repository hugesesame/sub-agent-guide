# Claude Code サブエージェント機能 — 会話要約

## 概要

Claude Code のビルトインサブエージェント機能について、メリット、活用判断基準、モデル指定方法を整理したドキュメント。

------

## 1. ビルトインサブエージェントのメリット

### 1.1 コンテキストウィンドウの保全

- メイン会話から探索と実装を分離
- 大量のログ・ファイル内容でコンテキストが肥大化するのを防止
- サブエージェントが独立したコンテキストで処理して要約だけを返す
- 長時間セッションで過去の指示忘れやノイズを防げる

### 1.2 権限の最小化と安全性

- 各エージェントに必要最小限のツール権限だけを割り当て可能
- 例：コードレビュー用は読み取り専用に制限
- 意図しないファイル変更・環境変数の書き換え防止

### 1.3 専門性による品質向上

- タスクを特定の領域に特化させた指示で実行
- コードレビュー専門：セキュリティ・パフォーマンス・可読性を徹底チェック
- デバッグ専門：エラーメッセージから根本原因を特定
- テスト専門：エッジケースまで網羅したテスト作成

### 1.4 並列実行による時間短縮

- 複数のサブエージェントを同時実行可能
- 例：フロントエンド調査 + バックエンド調査 + セキュリティチェックを並列実行
- 実践報告：作業スピードが体感で2〜3倍になる

### 1.5 コスト最適化

- タスクの性質に応じて最適なモデルを選択
- 例：grep処理が多い探索タスクは Haiku、複雑な判断は Opus
- トークン単価の削減と精度のバランスを取れる

------

## 2. ビルトインサブエージェント（自動起動）

Claude Code には以下の3種類が組み込まれており、Claude が文脈を読んで自動的に委譲：

| エージェント        | 用途                             | 特徴                           |
| ------------------- | -------------------------------- | ------------------------------ |
| **Explore**         | ファイル探索・コード検索         | 読み取り専用、高速             |
| **Plan**            | 計画立案・構想フェーズ           | 探索結果を別コンテキストで処理 |
| **General-purpose** | 複雑な操作・マルチステップタスク | 探索と実装の両方に対応         |

------

## 3. ウェブアプリケーション開発でのサブエージェント活用判断

### 3.1 最初から起動すべき場合 ✅

#### 複数工程が既に決まっている場合

```
実装フェーズが明確に定まった後：
  実装エージェント → コード生成
       ↓
  レビューエージェント（読み取り専用）→ セキュリティ・パフォーマンスチェック
       ↓
  テストエージェント → ユニットテスト・統合テスト
```

「実装 → レビュー → テストまで回して」と1回指示すれば、複数エージェントが並列で処理

#### コンテキストが「汚れる」と予見できる場合

- 大量のコード検索・解析が必要
- ログやファイルダンプが大量に出る
- データベースマイグレーション等の複雑な調査

これらは最初から Explore または Plan エージェントに任せると効果的

### 3.2 途中で切り替える場合 ⏱️

#### 不確実な初期段階では柔軟に対応

- **初期段階**：フレームワーク選定など方向性を詰める → メイン会話で OK
- **実装フェーズが定まった後**：明確なパイプラインが見える → エージェント投入
- **途中で複雑さが出た**：ファイルを何度も読み込みながら検討 → Explore エージェントに委譲

## 5. モデル指定方法

### 現在の理解レベル

サブエージェント定義時にモデルを指定可能：

- grep処理が多い探索 → **Haiku**（コスト重視）
- コードレビュー・テスト生成 → **Sonnet**（バランス型）
- 複雑な設計決定・マルチシステム統合 → **Opus**（精度重視）

### 指定方法の詳細

`.claude/agents/` 配下の Markdown frontmatter で指定するはず：

```markdown
---
name: your-agent-name
description: いつこのエージェントを呼ぶかの説明
tools: Read, Glob, Grep
model: ???  # ← 正確なフォーマットは要確認
---
```

**注記**：正確なフォーマット（`model: haiku` なのか `model: claude-haiku-4-5` なのか等）は、公式ドキュメント確認が必須

## 6. Theater/AV Production での活用例

Cruise ship theater system 開発での活用：

### 最初から定義すべき `.claude/agents/`:

```
✓ broadcast-config-reviewer
  役割：SMPTE, audio routing 仕様確認
  権限：読み取り専用
  モデル：Haiku（単純検索）

✓ dmx-integration-tester
  役割：各 lighting cue の検証
  権限：read, test execution
  モデル：Sonnet

✓ timeline-auditor
  役割：LTC との同期性確認
  権限：読み取り専用
  モデル：Haiku
```

### 活用パターン

ウェブUI部分の実装後、この「監査エージェント」群を並列投入：

- メイン会話を汚さずに品質チェック実行
- 各システム（SMPTE, DMX, MIDI, OSC）の連携検証を並列処理
- 複雑なワークフロー検証も効率化
